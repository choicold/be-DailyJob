### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 쿼리 스트링을 파싱하는 로직을 아예 클래스로 따로 뺐음.
- 그렇게 판단한 이유: 
  - HTTP Request 시 GET의 경우 헤더 첫 줄에 쿼리 스트링이 POST의 경우 HTTP Body에 쿼리 스트링이 오는 형태였음.
  - 기존 방식은 GET인 케이스에 대해 단일 처리를 진행하다보니 이 처리를 담당하는 클래스의 코드를 POST 시 HTTP Body 내의 쿼리 스트링을 파싱하는 데에도 사용해야했음.
- 고려했지만 선택하지 않은 대안: 
  - 기존 메서드를 public으로 바꾸고 해당 클래스에서 계속 가져다 사용하게끔 하고 돌려쓰는 방안을 고려했지만 선택하지 않았음. 
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 유지보수 측면에서 용이하게 확장함.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 
  - GET/POST 파라미터 처리 책임이 헷갈렸던 상황
- 처음 추측한 원인:
  - GET일 때와 POST일 때 파라미터 처리 로직을 기존 GET을 처리하는 객체에 종속되어 있다고 생각
- 원인을 확인하기 위해 시도한 방법:
  - 실제 POST 시 GET 객체에 파리미터 로직을 호출하여 User 객체에 값이 담기지 않음을 확인
- 실제 원인:
  - GET일 때와 POST일 때 파라미터 처리 로직을 기존 GET을 처리하는 객체에 종속되어 있어 코드 상의 로직 변경이 필요했음.
- 해결 방법:
  - 파라미터 처리를 해주는 클래스를 따로 분리하여 각 HTTP 메서드에 대응하는 파라미터 처리
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 클래스 설계 시 확장성을 확실히 고려해야겠다. 뭔가 클래스가 종속적이니까 수정 필요할 때 변경이 어렵다는 생각을 했음.
