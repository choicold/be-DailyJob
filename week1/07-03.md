# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 
  - 웹서버 3단계 GET으로 회원가입 나머지 부분
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 1주차 과제 리팩토링

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 
  - HTTP Request와 HTTP Response의 형태로 구성된 RequestHandler 코드를 각각 클래스로 분리
- 그렇게 판단한 이유:
  - 다양한 요청, 다양한 응답에 대응하고 집중하기 위해 기존 RequestHandler에 혼재되어 있던 책임을 분리할 필요가 있다고 생각함.
- 고려했지만 선택하지 않은 대안:
  - 처음부터 인터페이스 설계, 추상클래스 도입 등을 고려했지만 우선 생각할 수 있는 수준에서의 리팩토링을 해보고 또 그 다음 스텝으로 어떻게 도입하면 좋을지를 더 생각해보기로함.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 클래스의 책임을 분리했기 때문에 유지보수 및 테스트 측면에서 관리하기 용이하다고 생각함.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 기존 코드를 분리하는 과정에서 어느 정도 수준까지 구분하고 분리해야할지 막힘.
- 처음 추측한 원인:
  - 우선 크게 HttpRequest와 HttpResponse로 크게 나누기까지는 생각을 했지만 구현되어 있는 코드를 각각 어떻게 저 2개로 나눠야할지 어려움을 겪었던 것 같음.
- 해결 방법:
  - 너무나도 당연한 얘기일 수 있으나 요청을 받아 파싱 및 헤더 첫 줄을 저장하는 로직을 Request에 포함시킴.
  - 응답 헤더와 응답 본문에 대해 작성하고 리턴하는 로직을 Response에 포함시킴.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 리팩토링에 익숙하지 않다보니 겪은 문제로 어떠한 특정 문제라고 지칭하기보다는 경험 즉 숙련도의 문제라고 생각함.
  - 기존 코드를 계속 리팩토링해보며 경험을 쌓는 것을 방향으로 잡고 연습해야겠음.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 기존 코드를 리팩토링을 시도한 것
- 그 접근이 도움이 된 이유:
  - 어느 순간 코드를 짜고 있을 때 로직이 복잡해진다고 느끼거나 다른 성격의 함수들이 혼재되어 있다는 느낌을 받았음.
  - 이를 해결하려고 하고 실제 적용해보니 코드를 이해하기도 편하고 한결 기존 코드가 간결해진 느낌이 들었음.
- 다음에도 반복하고 싶은 점:
  - 학부때 배웠던 SOLID 원칙의 SRP가 왜 중요한지 몸소 체험하니까 직접 더 와닿는 느낌이라 이러한 방향성을 계속 유지하고 싶음.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 아쉬운건 아무래도 구현 시 막히고 막혔을 때 왜 막혔는지를 명확히 쪼개지 못한 접근이 아쉬움.
- 그렇게 된 원인:
  - 막힌 부분을 명확히 스스로에게 설명하지 못하고, 또 설명하더라도 내게 필요한 지식을 어떻게 어떠한 방향으로 습득해야할지 갈피를 제대로 잡지 못했음.
- 다음에는 다르게 해볼 구체적인 행동:
  - 막힌 부분이 있으면 생각이 닿는대로 뭐 IDE에 두서없이 작성해보거나 노션에 작성해보는 형태로 구체화해보고 싶음.
  - 필요한 지식은 최대한 AI에게 그때 그때 물어보는 방향이 웹서치보다는 훨씬 수월할듯 싶음. 
  - 물론 웹서치가 AI에 비해 비효율적인 것을 알지만 뭔가 AI를 이 과제할 때 최대한 덜 써보고 싶다는 바람에 그랬던 것 같음.
