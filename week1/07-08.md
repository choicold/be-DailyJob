# be-DailyJob

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
    - 세션-쿠키 기반 로그인 구현: 로그인 성공 시 UUID 기반 sessionId 생성 → SessionStore 저장 → 응답에 `Set-Cookie: SID=` 추가 → index.html 302 리다이렉트까지 동작
    - Set-Cookie 삽입 위치 결정: 로그인 성공 응답이 302이므로 found() 안에서 처리
    - 코드베이스 의존성 흐름 분석: RequestHandler가 WAS/Router/Controller/Service/Repository 접근까지 5개 역할을 겸하는 현황을 다이어그램으로 정리하고, 계층 분리 개선안(UserController → UserService → Database / SessionManager → SessionStore / StaticResourceHandler) 설계
    - 학습: 세션 ID에 UUID.randomUUID()를 쓰는 이유(SecureRandom 122비트 → 유일성 + 예측 불가능성), Map.of() 특성(불변, null 금지, 최대 10쌍 → 초과 시 ofEntries), 쿠키 `Path=/`의 의미
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
    - 설계한 구조 개선안의 실제 리팩토링
    - 로그인 실패 시 처리 방식 미결정, 아마도 토스트 팝업으로 진행할 것 같음.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 
  - HttpResponse에 sessionId를 직접 전달하지 않고, `Set-Cookie: SID=` 헤더 수준의 정보만 알게 함.
- 그렇게 판단한 이유: 
  - sessionId는 애플리케이션/도메인 개념이고 HttpResponse는 HTTP 프로토콜 계층 객체임.
    HttpResponse가 알아야 하는 건 세션이라는 내부 개념이 아님.
- 고려했지만 선택하지 않은 대안: 
  - sessionId를 그대로 넘기는 방식. 동작 관점에서는 문제가 없었지만 HTTP 계층이 도메인 개념에 의존하게 됨.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 
  - 계층 간 결합도 감소
  - 이후 세션 저장 방식이나 쿠키 정책이 바뀌어도 HttpResponse는 수정 불필요
  - 값을 넘겼을 때 받는 객체가 어떤 개념을 알게 되는가라는 재사용 가능한 판단 기준을 얻음.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 
  - 로그인 성공 응답에 Set-Cookie를 넣어야 하는데, HttpResponse 구조에서 어디에 어떻게 넣어야 할지 불명확했음
- 처음 추측한 원인: 
  - sessionId를 HttpResponse에 넘기고 내부에서 알아서 처리하면 된다고 생각
- 원인을 확인하기 위해 시도한 방법: 
  - 현재 응답 생성 코드에서 헤더와 본문의 경계를 확인
- 실제 원인: 
  - 응답 구조와 상태별 메서드의 역할을 정확히 잡지 않고 구현부터 시작함.
- 해결 방법: 
  - 로그인 성공 = 302 리다이렉트이므로 found()에서 Set-Cookie 헤더 추가
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 
  - 새 헤더 추가 시 HTTP 스펙상 어떤 응답에 붙는 헤더인지, 현재 코드에서 그 응답을 생성하는 지점이 어디인지부터 확인해볼 것 같음.
