### 1. 오늘 완료한 것

    - Figma에 있는 index.html, register.html, login.html, comment.html, write.html에 대한 요구사항을 이미지 업로드를 제외하고 충족시켰다.
    - JDBC Driver, DataSource, Connection Pool, HikariCP의 관계를 현재 코드와 연결해 학습했다.
  

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
    - 회원가입과 로그인 오류를 공통으로 표현할 오류 모듈의 최종 구조를 결정하지 못했다.
    - HikariCP 같은 커넥션 풀을 현재 프로젝트에 도입할지는 결정하지 않았다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
    - 좋아요 증가를 애플리케이션에서 조회하고 계산하지 않고, DB에서 원자적인 UPDATE로 처리하기로 했고, 어플리케이션 레벨에서 동시성 관련한 처리를 DB에 맡기기롤 결정
    - 중복 회원가입은 기존 MERGE를 사용하여 사용자를 덮어쓰는 대신 INSERT와 DB 제약조건으로 차단하기로 했음.
  

- 그렇게 판단한 이유:
    - 좋아요 수를 먼저 조회한 뒤 Java에서 +1하여 저장하면 동시 요청하면 증가하는 값이 유실될 수 있다.
    - 영속 DB에서 MERGE로 사용자를 덮어쓰면 기존 계정의 비밀번호까지 변경될 수 있어 보안 문제가 된다.
  

- 고려했지만 선택하지 않은 대안:
    - 좋아요 수를 조회한 뒤 Java에서 증가시켜 다시 저장하는 방식
  

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
    - 좋아요 증가가 한 개의 SQL로 표현돼 로직이 단순해지고 값 유실 가능성이 줄었다.
    - 회원 데이터의 무의식적인 덮어쓰기가 차단됐다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
    - H2 데이터베이스를 IntelliJ에서 확인하려고 연결했을 때 다음 오류가 발생했다.

      `Database may be already in use:
      "/Users/admin/Documents/be-was/data/be-was.mv.db"`

    - 또한 처음에는 H2 파일이 어디에 저장되는지, 애플리케이션이 JDBC를 통해 H2와 어떤 방식으로 연결되는지 명확히 이해하지 못했다.
  

- 처음 추측한 원인:
    - H2 설정이 잘못됐거나 파일이 손상됐을 가능성을 생각했다.
    - DataSource를 사용하고 있으므로 내부적으로 커넥션 풀도 사용하고 있을 수 있다고 생각했다.
  

- 원인을 확인하기 위해 시도한 방법:
    - 실제 JDBC URL과 H2 파일 경로를 - 프로젝트의 DataSourceFactory와 Repository가 Connection을 얻는 흐름을 살펴봤다.
    - IntelliJ DB 도구와 애플리케이션이 같은 DB 파일에 동시에 연결되는 상황을 확인했다.
    - DataSource, JDBC Driver, Connection Pool, HikariCP의 역할을 구분해 질문했다.
    - 서버를 재시작한 뒤에도 데이터가 유지되는지 E2E로 확인했다.
  

- 실제 원인:
    - 파일 기반 H2가 ./data/be-was.mv.db를 사용하고 있었고, 애플리케이션과 IntelliJ가 동일
      파일을 동시에 독점 방식으로 열려고 해 충돌한 것이었다.
    - DataSource는 Connection을 얻기 위한 표준 인터페이스일 뿐이며, DataSource를 사용한다고 해서 자동으로 커넥션 풀을 사용하는 것은 아니었다.
    - 현재 구조에는 HikariCP 같은 별도의 커넥션 풀 구현체가 적용되지 않았다.


- 해결 방법:
    - 애플리케이션이 DB 파일을 사용 중일 때는 연결을 종료한 뒤 IntelliJ에서 확인하는 방식으로 접근했다.

    - 현재 연결 구조를 다음과 같이 정리했다.

      Repository
      → DataSource
      → JDBC Connection
      → H2 JDBC Driver
      → 파일 기반 H2

    - 커넥션 풀이 필요하다면 HikariCP와 같은 별도의 DataSource 구현체를 추가해야 한다는 점을 이해했다.


- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - JDBC URL이 메모리 모드인지 파일 모드인지 확인한다.
    - 실제 DB 파일 경로와 현재 작업 디렉터리를 확인한다.
    - 같은 DB 파일을 사용하는 다른 애플리케이션이나 DB 도구가 실행 중인지 확인한다.
    - DataSource의 실제 구현 클래스가 무엇인지 확인한다.
    - Connection 생성과 종료가 어디에서 이루어지는지 코드로 추적한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
    - 기능이 동작하는지만 확인하지 않고 JDBC와 H2가 실제 코드에서 연결되는 원리를 질문했다.
    - DataSource와 Connection Pool이 같은 개념인지, HikariCP가 Spring에 종속된 것인지 단계적으로 질문했다.
    - AI가 제시한 객체 관계 그림이 현재 코드베이스와 맞지 않는다는 점을 지적했다.
    - 중복 회원가입이 영속 환경에서는 기존 계정 덮어쓰기와 계정 탈취 문제로 이어질 수 있다는 점을 확인했다.
    - 좋아요 동시성 문제를 복잡한 락부터 도입하지 않고 DB의 원자적 갱신으로 해결할 수 있는지 검토했다.


- 그 접근이 도움이 된 이유:
    - 라이브러리 사용법이 아니라 계층과 책임을 이해하게 됐다.
    - 좋아요 기능을 요구사항보다 과도하게 복잡하게 만들지 않으면서도 기본적인 동시성 문제를 줄였다.
  

- 다음에도 반복하고 싶은 점:
    - 설명을 들은 뒤 반드시 실제 파일, 생성자 의존성, SQL, 테스트로 확인한다.
    - 기능 구현 전에 데이터 흐름과 책임이 놓일 계층을 먼저 질문한다.
    - 예외가 발생하면 “어디에서 발생하고, 어디에서 의미를 바꾸며, 사용자에게 어떻게 전달되는가?”를 추적한다.
    - 여러 대안이 있다면 선택 이유뿐 아니라 포기한 대안과 단점도 기록한다.
